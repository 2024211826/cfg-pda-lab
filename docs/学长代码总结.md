本次阅读只做静态检查，没有运行参考程序。

## 2. lab1 代码简评

`lab1.cpp` 实现的是 epsilon-NFA/NFA 到 DFA 的转换，核心结构如下：

- `NFA`：状态集合、输入符号集合、转移表、初始状态、终态集合。
- `DFA`：DFA 状态集合、输入符号集合、转移表、初始状态、终态集合。
- `eClosure()`：通过 BFS 计算 epsilon 闭包。
- `dfaStateFromNfaStates()`：从 NFA 状态集合和输入符号计算 DFA 新状态。
- `convertNFAtoDFA()`：用子集构造法生成 DFA。

这一部分与当前作业关系不大，因为当前作业关注 CFG 化简和 PDA 到 CFG。不过它的代码风格有两个可借鉴点：

- 使用 `std::set` 表示状态集合，可以自然去重和排序。
- 使用 `std::map<pair<...>, ...>` 表示转移表，便于按“当前状态 + 输入符号”查找。

需要注意的是，`lab1.cpp` 中包含 `windows.h` 和 `Sleep()`，这会降低跨平台性。它只是界面展示效果，不适合放到正式实验核心代码中。

## 3. lab2 代码结构

`lab2` 由两部分组成。

第一部分是 CFG 化简，位于 `CFG_Simplify.h` 和 `CFG_Simplify.cpp`：

- `eli_epsilon()`：消除 epsilon 产生式。
- `eli_single()`：消除单产生式。
- `eli_useless()`：消除无用符号。
- `printout()`：输出化简后的文法。

第二部分是 PDA 到 CFG，位于 `PDAtoCFG.cpp`：

- `PDA` 结构体：保存状态集合、输入符号集合、栈符号集合、转移函数、初始状态、初始栈符号和接受状态集合。
- `CFG` 结构体：保存非终结符集合、终结符集合、产生式集合和开始符号。
- `PDAtoCFG::convert()`：构造 PDA 对应 CFG。
- `generateProductions()`：根据压栈串生成 CFG 产生式。
- `removeUnusedSymbols()`：尝试删除 PDA 转 CFG 后的无用符号。
- `transfer()`：把 `[q,z,r]` 形式的复杂非终结符重命名成单字符，以便复用 CFG 化简函数。
- `main()`：通过模式选择执行 CFG 化简或 PDA 到 CFG。

## 4. 可借鉴点

### 4.1 CFG 化简流程

`lab2` 的整体流程与课件和作业要求基本一致：

```text
消除 epsilon 产生式
消除单产生式
消除无用符号
```

这种顺序可以参考。前两步可能引入新的冗余产生式，最后统一删除无用符号更合理。

### 4.2 数据结构选择

CFG 部分使用：

```cpp
set<char> N;
set<char> T;
map<char, set<string>> P;
char S;
```

这种结构对“单字符非终结符”的简单 CFG 样例比较方便。`set` 可以自动去重，`map` 可以按左部归类产生式。

PDA 部分使用：

```cpp
set<State> Q;
set<Terminal> T_PDA;
set<StackSymbol> Gamma;
map<tuple<State, Terminal, StackSymbol>, set<pair<State, string>>> delta;
```

这种结构比 CFG 部分更通用，状态、终结符和栈符号都用字符串表示，适合表达 `q0`、`q1`、`z0` 这类多字符符号。

### 4.3 PDA 到 CFG 的理论框架

`PDAtoCFG::convert()` 中采用了课件里的基本构造：

- 新建开始符号 `S`。
- 对每个 `q in Q` 加入 `S -> [q0,z0,q]`。
- 为每个三元组 `(q, z, r)` 构造非终结符 `[q,z,r]`。
- 对每条 PDA 转移生成对应 CFG 产生式。

这个方向是正确的，可以作为理解 PDA 到 CFG 的参考。

## 5. 主要风险点

### 5.1 没有传统内存泄漏，但有未定义行为风险

源码没有使用 `new`、`delete`、`malloc`、`free`，主要依赖 STL 容器自动管理内存。因此传统意义上的内存泄漏风险不明显。

更大的问题是越界访问、迭代器失效和错误假设。

### 5.2 PDA 压栈串处理存在越界访问

`PDAtoCFG.cpp` 中 `generateProductions()` 直接访问：

```cpp
stackStr[0]
stackStr[1]
stackStr[2]
```

但函数没有先判断 `stackStr` 的长度。如果压栈串长度为 1，访问 `stackStr[1]` 就越界；如果长度为 2，访问 `stackStr[2]` 也越界。

这说明代码实际上只适配非常特定的输入形式，不适合直接用于通用 PDA 到 CFG 转换。

### 5.3 PDA 到 CFG 只处理部分压栈长度

学长报告的“现存问题”中也写明：无法识别输入为单个或大于等于三个栈中符号的情况，实际限制为压栈 0 个或 2 个符号。

这和课件中一般形式 `B1B2...Bk` 不一致。当前作业给出的 PDA 恰好有 `Bz0`、`BB` 和 `epsilon`，长度主要是 2 或 0，但最终代码最好不要把这种特殊性写死。

### 5.4 删除容器元素的迭代器写法不安全

`PDAtoCFG.cpp` 中有类似写法：

```cpp
for (auto it = cfg.N_CFG.begin(); it != cfg.N_CFG.end(); ++it)
    if (used.find(*it) == used.end())
        it = cfg.N_CFG.erase(it);
```

`erase()` 已经返回下一个迭代器，`for` 循环末尾又会执行 `++it`，可能跳过元素，严重时可能对 `end()` 继续递增。

`cfg.P_CFG.erase(it)` 的循环也有同类问题。

### 5.5 遍历集合时继续插入同一集合，逻辑不稳

`removeUnusedSymbols()` 中遍历 `unused` 的同时又向 `unused` 插入新的元素：

```cpp
for (const auto& unusedState : unused) {
    ...
    unused.insert(it->first);
}
```

`std::set` 插入通常不会让已有迭代器失效，但这种写法逻辑上不清晰：新插入的元素是否会被本轮遍历到依赖容器遍历过程，不适合作为严肃的闭包计算。更稳妥的做法是使用队列，或者循环到集合不再变化。

### 5.6 `transfer()` 的重命名方案不可靠

`transfer()` 把复杂非终结符如 `[q0,z0,q1]` 强行重命名为单个字符 `A`、`B`、`C` 等。

问题包括：

- 只能支持有限个单字符非终结符，复杂 PDA 很容易不够用。
- 可能与已有非终结符或终结符冲突。
- 替换时只处理 `find()` 找到的第一个位置，右部若出现多个同一非终结符，可能替换不完整。
- 用字符串包含关系查找复杂非终结符，可能出现误匹配。

这个函数是 `lab2` 中最不适合照搬的部分之一。

### 5.7 CFG 化简只适合单字符符号

`CFG_Simplify.cpp` 的接口把非终结符和终结符都设计为 `char`：

```cpp
set<char> N;
set<char> T;
map<char, set<string>> P;
char S;
```

这会导致它不适合直接处理 `[q,z,r]` 这种多字符非终结符。因此学长代码才需要 `transfer()` 做强制重命名。

如果我们重新设计，最好从一开始就使用 `string` 或结构化符号表示 CFG 符号，避免后续重命名带来的不稳定性。

### 5.8 新开始符硬编码为 `T`

`CFG_Simplify.h` 中写死：

```cpp
#define NEW_STARTER 'T'
```

`eli_epsilon()` 中又写：

```cpp
S = NEW_STARTER;
P1[NEW_STARTER] = {"#", "S"};
```

这里有两个问题：

- `T` 可能已经是已有非终结符或终结符。
- `{"#", "S"}` 写死旧开始符是 `S`，如果原开始符不是 `S`，结果会错。

### 5.9 输入校验不足

代码中存在多处直接取下标的写法：

```cpp
inputNtmp[0]
sign[0]
it->first[0]
```

如果输入为空字符串，就可能出错。模式选择、PDA 转移格式、产生式格式也缺少系统性的校验。

### 5.10 源码注释编码混乱

`CFG_Simplify.cpp` 中部分中文注释显示为乱码，说明文件编码和读取环境不一致。这不影响编译，但影响维护和报告整理。

## 6. 与当前作业的差异

### 6.1 lab1 与当前作业无直接关系

`lab1` 是 NFA 到 DFA，当前作业是 CFG 化简和 PDA 到 CFG。可以借鉴 STL 表示集合和转移表的方式，但不应把它作为当前实验的核心参考。

### 6.2 lab2 的 PDA 测试样例符号方向不同

当前 `docs/homework.docx` 给出的 PDA 是：

```text
delta(q0,b,z0) = {(q0,Bz0)}
delta(q0,b,B)  = {(q0,BB)}
delta(q0,a,B)  = {(q1,epsilon)}
delta(q1,a,B)  = {(q1,epsilon)}
delta(q1,epsilon,B)  = {(q1,epsilon)}
delta(q1,epsilon,z0) = {(q1,epsilon)}
```

学长报告中的测试输入使用的是先读 `a` 压栈、再读 `b` 弹栈：

```text
q0 a z0 q0 Az0
q0 a A  q0 AA
q0 b A  q1 #
q1 b A  q1 #
q1 # A  q1 #
q1 # z0 q1 #
```

二者结构相似，但终结符方向相反，栈符号也从 `B` 换成了 `A`。如果用学长样例做验证，需要改回当前作业给定的 `b/a/B/z0` 版本。

### 6.3 学长报告中的输入输出格式可参考，但应重新设计

学长报告的 CFG 输入格式是：

```text
SABCD
abcd
S a|bA|B|ccD
A abB|#
B aA
C ddC
D ddd
@
```

这个格式直观、易手输，可以参考。

但若最终程序支持多字符非终结符，单纯把产生式右部当作字符序列会不够用。输入格式需要与最终数据结构保持一致。

## 7. 建议的使用方式

可以参考：

- CFG 化简的整体流程。
- 使用集合迭代到不动点的思路。
- PDA 到 CFG 的非终结符构造框架。
- 报告章节组织方式。
- 控制台输入输出样例的组织方式。

不建议直接复用：

- `transfer()` 的复杂非终结符重命名。
- `generateProductions()` 对压栈串长度的硬编码处理。
- `removeUnusedSymbols()` 中不稳定的集合修改逻辑。
- 以 `char` 为核心的 CFG 数据结构。
- 硬编码新开始符 `T` 和旧开始符 `S` 的处理。
- 缺少输入校验的交互逻辑。

总体判断：`学长/lab2` 可以作为理解实验任务和报告格式的参考，但不适合作为最终代码直接提交。若要保证程序稳定、可解释、方便测试，建议重新实现核心数据结构和 PDA 到 CFG 转换逻辑，只保留其算法方向作为参考。
